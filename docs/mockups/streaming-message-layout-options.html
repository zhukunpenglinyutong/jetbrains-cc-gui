我的建议

优先选方案 C 的修正版：不是“智能阶段分组”，也不显示任何编号式任务外标签。 它只通过轻量视觉分隔保持阅读节奏，并按工具事件顺序展示真实内容。

  • 实时阶段：每次识别到新输出边界，就在响应容器内部追加一个无标题内容组。
  • 标题生成：只根据工具调用做结构统计，例如“读取文件 1”“编辑文件 1”“新增文件 1”。
  • 完成阶段：最终快照同步时保留内容组边界，合并只发生在同一内容组内。
  • 历史回显：有内容组边界的新消息按该结构渲染；没有边界的旧历史继续走现有合并逻辑。

为什么不做语义阶段

长任务里工具调用会交错出现：读取文件、编辑文件、再次读取、再编辑、新增文件、跑测试都可能重复多轮。 如果前端强行归类成“定位”“修改”“收尾”，很容易出现错误标题。

结构统计更稳定，因为它来自工具调用本身。用户能看到真实顺序，也能快速扫出每段内容发生了什么。
